# AI Assistance Log

Record every instance where you use AI to generate, refactor, or debug code. Four fields per entry: what you asked for, what the AI produced, what you changed, and why you changed it.

Document **3+ entries** in your AI_LOG covering **3-5 distinct patterns** from this list -- planning, multi-agent critique, plan-mode scaffolding, refactor, test generation. Aim for variety: an entry that just says "asked Claude to write X" three times doesn't demonstrate AI as a real engineering tool.

## Instance 1: Planning the build (interactive HTML plan / mission-control hub)

| Field | Detail |
|-------|--------|
| **Date** | 2026-06-03 |
| **Task** | Use Claude's plan mode to scope the whole hackathon build and produce a shared plan we could all engage with. We asked for a single HTML file we could open in a browser, make decisions in, and export as deterministic JSON to feed back to Claude. |
| **What AI Generated** | Claude explored the starter repo and the GOV.UK Design System, then produced a self-contained, interactive HTML "mission-control" page: the six MVP floor items, a GOV.UK pattern reference, a concrete (fictional) eligibility ruleset with worked test cases, a work-division strategy chooser, a task board with owner dropdowns, stretch goals, an AI_LOG plan, and a "Generate JSON" button that emitted all our choices deterministically. After we answered a few scoping questions (timebox, scope, ruleset) it updated the plan and re-rendered the HTML. |
| **What You Changed + Why** | The plan was genuinely engaging to look at and it made us home in on key decisions fast — we chose a work-division strategy and a first page to build, and Claude re-rendered the HTML to match. But two problems led us to drop most of it. **(1) Forward dependencies.** Many sections asked for input whose meaning depended on answers we hadn't given yet. Example: Section 3 asked which page to build first, but the CLAUDE.md starter in Section 4 directly below had already hard-coded "Property type", so it was unclear whether our choice mattered; likewise the task board's suggested owners depended on a strategy chosen in another section, and the AI_LOG plan asked us to assign five patterns across four developers before we'd even entered our names. **(2) Wrong order — cart before the horse.** More fundamentally, the plan front-loaded design and content decisions (the eligibility rules, GOV.UK patterns, which page to build first) when what we actually needed *first* was to agree a way of working as a team and the order in which to tackle the work ahead. That process decision mattered more than the design decisions, but it was buried among them. As an inexperienced team without a strong structure of our own, being shown every decision at once was overwhelming; we needed to take things one step at a time and weren't set up to impose that sequence ourselves — so we dropped most of the plan and worked from the chat. **Future improvement:** sequence the plan — settle *how we work and in what order* before surfacing any design/content decisions — and make it branch interactively, revealing only the next relevant decision based on earlier answers, so each step shows just the information needed to make that one choice. |

## Instance 2: Agreeing a clash-free way of working + a shared progress checklist

| **Task** | Having dropped the first all-in-one plan (Instance 2) for front-loading design decisions, we asked Claude to help us agree a *way of working* first: a concrete task split for four people that wouldn't cause merge conflicts, plus a simple shared checklist we could track progress on and show in our presentation. |
| **What AI Generated** | Claude read the current repo state (found `eligibility.js` and tests missing, the Check-answers and Result pages still stubs, and `App.jsx` as the one file every task would touch), then proposed a non-overlapping 4-way split where each person owns separate files. It identified the key risk — three tasks all editing `App.jsx` — and recommended a one-off "wiring commit" merged *first* so `App.jsx` is frozen before parallel work starts. It produced `TEAM_PLAN.md` and a self-contained GOV.UK-styled `team-plan.html` with tickable checkboxes that persist in the browser and a live progress bar, defined the exact `checkEligibility` return contract so the eligibility and result tasks could run in parallel, scaffolded the branches (wiring; eligibility with 7 passing tests), and opened the PRs. |
| **What You Changed + Why** | We deliberately ran this the opposite way to Instance 2 — **process before design**. This plan settled *how we'd work and in what order* before any content decisions, which fixed the "cart before the horse" problem from last time and was far less overwhelming for an inexperienced team to act on. **(1)** We kept the eligibility *policy* Claude generated only as a placeholder — flagged as a fictional first draft for Person A to rewrite, since the scheme rules are our decision, not the AI's; we kept the function's return *shape* because the Result page depends on it. **(2)** The plan proved itself in practice: when a teammate merged Check-answers work into `main`, it collided with the wiring commit's `App.jsx` edits *exactly* as the plan predicted. Claude rebased and resolved the conflict (keeping the colleague's check-answers line, keeping our `PhaseBanner` + Result wiring), which confirmed the "merge wiring first, then freeze `App.jsx`" rule was right. The lesson we'd apply next time: actually merge the wiring PR *before* anyone touches `App.jsx`, and there'd have been no conflict at all. |

## Instance 3: Verifying the result-page UI against the GOV.UK source (research → refactor)

| Field | Detail |
|-------|--------|
| **Date** | 2026-06-04 |
| **Task** | Decide how the eligibility Result page should present outcomes (eligible / partial / not-eligible). We were unsure whether to use the green confirmation panel, a grey "not eligible" panel, or neither, and asked Claude to research the GOV.UK Design System and tell us what conforms to best practice. |
| **What AI Generated** | Claude first built the Result page with a green `govuk-panel` for "eligible" and (at our request) a grey `govuk-panel--not-eligible` for partial/not-eligible. When we questioned it, Claude fetched and quoted three GOV.UK source pages — the Panel component, the Confirmation pages pattern, and the "Check a service is suitable" pattern — and reported: (a) the confirmation panel is only for *completed transactions*; (b) GOV.UK Frontend has no grey/negative panel variant, so our `--not-eligible` class was invented, not real; (c) the eligibility-checker pattern prescribes a plain page heading + body for every outcome. |
| **What You Changed + Why** | We removed **both** panels and now render every outcome as a plain `govuk-heading-xl` `<h1>` + reason + next steps, per the "Check a service is suitable" pattern. We also deleted the invented `.govuk-panel--not-eligible` CSS and rewrote the Result guidance in CLAUDE.md to cite the pattern, so the team stops flip-flopping on it. **Why:** the green panel would have misrepresented a preliminary eligibility check as a completed transaction, and the grey variant was not a real GOV.UK component — having AI verify against the primary source (rather than trust its first answer, or ours) overturned an assumption we would otherwise have shipped. The lesson: treat AI's first design choice as a draft and make it cite the source. |
