1Overview & the MVP floor

Build a digital service that replaces a 35-minute eligibility phone call. Citizens answer 5 questions, one per page, and learn instantly whether they qualify for the Green Home Grant, what for, and what to do next.

Win condition: judges reward a clean, complete, accessible service over a flashy incomplete one. Ship the six floor items below first — then build outward. Document your AI use as you go, not at the end.

The six mandatory items (the floor)

  • MVP 1Multi-page question flow — start page → 5+ question pages, one thing per page.
  • MVP 2Check-your-answers — summary list with a “Change” link for every answer.
  • MVP 3Result page — clear outcome: eligible / partial / not eligible.
  • MVP 4GOV.UK components — correct typography, spacing, heading hierarchy (use the existing App.css classes).
  • MVP 5WCAG 2.2 AA — labelled inputs, keyboard nav, 4.5:1 contrast, visible focus.
  • MVP 6AI usage log — AI_LOG.md, 3+ entries, 4 fields each, varied patterns.

2Architecture decisions (and the why)

The starter already has all 9 page routes, working router, and GOV.UK CSS. Lock these four decisions before anyone writes a feature — they kill the two failure modes that sink hackathon teams.

1. One state object, lifted to App.jsx

Hold a single formData object in useState at the top of App.jsx; pass it and an updateField(key, value) callback down to each page via props.

Why: the #1 hackathon failure is “form state lost between pages.” One source of truth fixes it for good.

2. Eligibility = a pure function

Put the logic in src/utils/eligibility.js — input: formData, output: a result object. No React inside it.

Why: testable without rendering. This is how you hit “5+ unit tests” cheaply and how you keep logic out of components.

3. Reuse the CSS — don’t write new CSS

Use govuk-heading-xl, govuk-body, govuk-button, govuk-radios, govuk-form-group--error, etc. They’re already in App.css.

Why: custom CSS burns time and breaks contrast/spacing compliance. The scale is already correct.

4. Mount what’s already built

PhaseBanner and GovukFooter exist but aren’t mounted. Add them to the layout in App.jsx. Footer links to the accessibility statement.

Why: free marks — phase banner + footer signal GOV.UK fluency to judges.

Architecture contract (agree this out loud before splitting): answer keys are propertyType, ownership, income, insulation, heating; each is a string from the option list in §4; updateField is the only way to mutate state; navigation uses useNavigate with the existing routes.

Start page

One thing per page (question pages)

The question is the page <h1>, set via the <legend> (radios) or <label> so screen readers hear it once. A back link at the top; a left-aligned “Continue” button (not “Next”). Offer “I do not know” where it’s honest.

govuk-summary-list with key / value / action rows. Every row’s “Change” link has visually-hidden text naming the field (<span class="govuk-visually-hidden">property type</span>). Final button: “Accept and continue”. Change links return the user to that question with the value pre-filled.

Validation & errors

On error: prefix the page <title> with “Error: ”, show an error summary (role="alert", “There is a problem”, focus it on load) linking to each bad field, plus an inline error next to the field with the same wording. Use govuk-form-group--error.

Result / confirmation

A clear outcome up top. For “eligible” use a green govuk-panel; for partial/not-eligible state it plainly and explain why and what to do next. Never rely on colour alone — back it with text.

Decision logic

export function checkEligibility({ propertyType, ownership, income, insulation, heating }) {
  // 1. Gate — private renters must apply via their landlord
  if (ownership === 'private-rent')
    return { outcome: 'not-eligible', funds: [], reasons: ['landlord-must-apply'] };

  // 2. What could be funded
  const needsInsulation   = insulation !== 'both';
  const insulationUnknown = insulation === 'dont-know';
  const canHeatPump       = heating !== 'heat-pump' && propertyType !== 'flat'; // flats: no external unit
  const funds = [];
  if (needsInsulation) funds.push('insulation');
  if (canHeatPump)     funds.push('heat-pump');

  // 3. Nothing left to fund
  if (funds.length === 0)
    return { outcome: 'not-eligible', funds: [], reasons: ['already-efficient'] };

  // 4. Income sets the funding level
  if (income === 'under-31000')
    return { outcome: 'eligible', funds, fundingLevel: 'full', survey: insulationUnknown };

  if (income === '31000-50000')
    return { outcome: 'partial', funds, fundingLevel: 'contribution', survey: insulationUnknown };

  // income === 'over-50000' — insulation not funded; heat-pump voucher only
  if (funds.includes('heat-pump'))
    return { outcome: 'partial', funds: ['heat-pump'], fundingLevel: 'voucher', survey: false };

  return { outcome: 'not-eligible', funds: [], reasons: ['income-too-high'] };
}

9AI_LOG plan

The brief wants variety — 3+ entries across 3–5 distinct patterns, not “asked Claude to write X” three times. Assign each pattern to whoever will own that piece of work so the log writes itself.

Each entry needs 4 fields: Date · Task · What AI Generated · What You Changed + Why. The “What You Changed + Why” is where marks live — show judgement, not just copy-paste. Use the seeded AI_LOG.md.
